# PR: 텔레그램 주차 조회/결제 정합성 개선 및 주차 이력 UI 자동 갱신

## 배경

기존에는 텔레그램 조회/결제 흐름과 실제 주차 DB 처리 기준이 서로 맞지 않는 부분이 있었습니다.

- 최근 이용 내역이 `camera` 1개 채널만 조회되거나 번호판 부분 일치로 섞일 수 있었음
- 결제 버튼이 `plate` 기준으로 동작해, 재입차나 동일 번호판 다중 이력에서 오반영 위험이 있었음
- 주차 입출차와 활성 레코드 연결이 `objectId` 중심이라 추적 ID 변경 시 레코드 정합성이 흔들릴 수 있었음
- 수동 번호판 수정 후 DB는 바뀌어도 주차 이력/차량 정보 테이블 UI가 바로 갱신되지 않았음
- 텔레그램 표시 요금과 실제 출차 정산 금액의 계산 정책이 달랐음

이번 변경은 이 흐름들을 하나의 기준으로 정리해, 사용자에게 보이는 정보와 실제 저장되는 데이터가 더 일관되도록 맞추는 데 목적이 있습니다.

## 주요 변경 사항

### 1. 텔레그램 최근 이용 내역을 4채널 통합 + 번호판 정확 일치로 변경

- `최근 이용 내역` 조회가 더 이상 `camera` 단일 채널에 고정되지 않음
- 전체 `parking_logs`에서 등록된 차량번호와 정확히 일치하는 레코드만 조회
- 출차 완료된 건만 필터링해 최신 3건 표시
- 메시지에 채널 정보도 함께 표시

## 2. 텔레그램 요금 조회를 현재 주차 기준으로 정리

- 우선 현재 주차 중인 활성 레코드를 4채널 전체에서 조회
- 현재 주차 중이면 `entry_time ~ now` 기준으로 실시간 예상 요금 표시
- 현재 주차 중인 내역이 없을 때만 최근 미납 정산 건을 안내
- 미납 건 안내에서는 DB에 저장된 `total_amount`를 그대로 사용

### 3. 결제 흐름을 `recordId` 기준으로 고정

- 텔레그램 결제 버튼 payload를 `PAY_<plate>`에서 `PAYREC_<recordId>`로 변경
- 결제 콜백 시 해당 출차 완료 레코드 1건만 조회/반영
- `exit_time`이 있고 `정산대기` 상태인 레코드만 결제 가능 대상으로 처리
- 기존 plate 기반 구버튼은 만료 처리하고 재조회 유도

### 4. 주차 레코드 연결 기준을 `reidId` 우선으로 정리

- 활성 주차 레코드 조회, 입차 중복 방지, 번호판 업데이트, 출차 처리에서 `reidId`를 우선 사용
- `objectId`는 현재 추적 객체를 가리키는 참고값으로만 유지
- 카메라 측 객체 ID가 바뀌더라도 같은 차량 레코드를 더 안정적으로 이어가도록 구조를 정리

### 5. 수동 번호판 수정 후 관련 UI 자동 갱신

- 차량 쪽에서 수동 번호판 입력이 성공하면 활성 주차 레코드의 번호판도 같이 갱신
- 이후 주차 이력 테이블이 자동 새로고침되도록 연결
- 차량 정보 DB 테이블도 함께 자동 새로고침되도록 연결

### 6. 주차 요금 정책 단일화

- `parking::calculateParkingFee(...)`를 유일한 요금 계산 함수로 사용
- 텔레그램 현재 요금 조회, DB 화면 표시 요금, 출차 시 최종 정산 금액이 모두 같은 정책을 따르도록 정리
- 저장소 내부에 남아 있던 별도 출차 정산 계산 함수는 제거
- 텔레그램 출차 안내 문구도 현재 정책 기준인 `5분 이내 무료`로 수정

## 영향 범위

- 텔레그램 사용자 조회/결제 UX
- 주차 입출차 기록의 레코드 연결 정합성
- 주차 이력 테이블 및 차량 정보 DB 테이블의 갱신 타이밍
- 실제 정산 금액과 UI/텔레그램 표시 금액의 일관성

외부 API 형식이나 DB 스키마 변경은 없습니다.

## 확인 방법

1. 텔레그램에 등록된 차량번호로 `최근 이용 내역` 조회
2. `camera ~ camera4` 전체 채널에서 해당 번호판과 정확히 일치하는 출차 완료 이력이 최신순으로 보이는지 확인
3. 현재 주차 중인 상태에서 `요금 조회` 시 현재 요금이 실시간으로 계산되어 보이는지 확인
4. 현재 주차 중이 아닌 상태에서 미납 출차 건이 있으면 `미납 정산 안내`와 결제 버튼이 보이는지 확인
5. 결제 버튼 클릭 시 해당 `recordId` 한 건만 `결제완료`로 반영되는지 확인
6. 차량이 재입차한 뒤 예전 결제 버튼을 눌러도 새 활성 건이 갱신되지 않는지 확인
7. 수동 번호판 수정 후 주차 이력 테이블과 차량 정보 DB 테이블이 자동으로 갱신되는지 확인
8. 출차 직후 저장된 `total_amount`와 텔레그램/DB 화면 계산 기준이 동일한지 확인

## 리뷰 포인트

- `reidId` 우선 연결이 입차/출차/번호판 보정 흐름 전반에서 일관되게 적용되는지
- 텔레그램 결제 콜백이 출차 완료 레코드만 안전하게 갱신하는지
- 4채널 통합 조회 시 최근 이용 내역/요금 조회가 의도한 우선순위로 동작하는지
- 수동 번호판 수정 후 UI 자동 새로고침이 기존 검색 상태를 깨지 않는지
- 주차 요금 계산이 현재 정책 함수 하나로만 남아 있는지

## 비고

- 이 변경은 정합성 개선이 목적이며, 과거에 이미 저장된 `total_amount`를 재계산하는 마이그레이션은 포함하지 않습니다.
- 현재 환경에서는 `nmake` 기반 빌드를 끝까지 수행하지 못해 전체 컴파일 검증은 별도로 필요합니다.
